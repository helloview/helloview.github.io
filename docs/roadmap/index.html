掌握技术栈的 5 个要点​

  • 技术产生的背景、来龙去脉、技术解决行业的什么问题
  • 技术基本用法、性能、最佳实践、核心原理及实现（进阶可以是源码阅读，代码实现）
  • 个人如何学习技术，在什么项目中如何使用技术或做了什么解决方案，并拿到了一个什么技术/业务的结果
  • 同生态：技术生态中有没有什么竞品或同类解决方案，各自的优缺点和适用场景是什么
  • 不同时空：技术发展的态势，个人的观察、解读、预判

Node.js & Express.js​

  • Node.js基础及与其他服务器端技术的区别：了解Node.js是什么以及它与其他服务器端技术的不同之处，特别是它基于JavaScript并能够进行客户端和服务器端的开发
  • Express.js中的中间件：了解中间件在Express.js中的功能，它位于请求和响应对象之间，执行如日志记录、身份验证和错误处理等任务
  • Node.js事件循环：熟悉Node.js事件循环，这是一个处理异步操作的机制。它允许Node.js同时管理多个请求和操作
  • Express.js中的HTTP方法：理解Express.js中HTTP GET和POST方法的区别，GET用于从服务器检索数据，而POST用于向服务器发送数据
  • Node.js模块系统：了解Node.js模块系统对于在应用程序中组织和重用代码的重要性
  • Express.js路由器：理解Express.js路由器的目的，它用于处理不同的路由和URL
  • Node.js中的package.json文件：了解Node.js应用程序中的package.json文件，其中包含有关应用程序及其依赖项的信息
  • Node.js中的子进程和工作线程：了解Node.js中子进程和工作线程的区别及各自的用例
  • Node.js REPL：熟悉Node.js REPL（读取-评估-打印-循环），这是一个用于测试和实验Node.js代码的交互式控制台
  • Express.js的视图引擎：了解Express.js的视图引擎的用途以及如何用它来渲染视图

Hybrid APPs​

  • 开发过 React Native app
  • Webview 性能优化：优化 WebView 的性能是至关重要的，涉及到页面加载、交互无反馈、页面白屏和数据加载等多个阶段
  • App 和 H5 的通信：涉及到 H5 端调用原生 App 端方法和原生 App 端调用 H5 端方法。Android 和 iOS 在这方面的实现有所不同。例如，Android 使用 loadUrl 和 evaluateJavascript 方法，而 iOS 使用 evaluateJavaScript 和 stringByEvaluatingJavaScriptFromString 方法
  • JSBridge 桥接器：JSBridge 是一种通信机制，允许原生和 H5 之间进行双向通信。它主要通过 JS 引擎或 WebView 容器进行协议约定。其中，有两种主要的实现方案：拦截 URL Schema 和 WebView 容器注入 JSAPI

XSS攻击和防御:​

  • XSS的原理：XSS（跨站脚本攻击）指攻击者通过将可执行代码注入网页中，这些恶意代码与网站的正常代码混合，导致浏览器无法区分哪些脚本是可信的
  • 防御措施：包括对数据进行严格的输出编码，如HTML元素的编码、CSS编码、JS编码、URL编码等。还可以使用CSP HTTP Header（内容安全策略）来增加攻击难度

CSRF攻击和防御:​

  • CSRF的原理：CSRF（跨站请求伪造）是一种利用用户在网站上已经获得的认证来冒充用户执行操作的攻击
  • 防御措施：包括设置Cookie的SameSite属性，进行同源检测，以及使用CSRF Token或双重Cookie验证机制

前后端数据加密:​

  • 重要性在于确保数据在传输过程中的安全性，常用的方法包括使用HTTPS协议进行加密通信
  • 对于敏感数据，前端在发送前应进行加密处理，并在后端进行相应的解密

Service Worker的使用:​

  • Service Worker可以用来定义缓存策略，例如请求成功后，将结果存入caches
  • 可以用来处理失败的请求，比如尝试从localStorage中读取先前成功的请求数据

用户输入的处理:​

  • 永远不要信任用户的输入。对用户输入进行校验，如使用正则表达式，限制长度，对特殊字符进行转换等

X-Frame-Options​

  • X-Frame-Options响应头是一个安全特性，用于帮助避免点击劫持攻击。它指示浏览器是否允许渲染一个页面在<frame>, <iframe>, <embed>或<object>中。常见的设置值包括:
    • DENY：不允许页面在任何框架中显示
    • SAMEORIGIN：页面只能在相同源域下的框架中显示

Referrer-Policy​

  • Referrer-Policy头部控制HTTP Referer头部的内容，以及在何种情况下发送它。这有助于保护页面的来源信息，减少隐私泄露的风险。常见的策略包括：
    • no-referrer：不发送Referer头部
    • no-referrer-when-downgrade：在从HTTPS到HTTP降级时不发送Referer头部（默认行为）
    • origin：只发送来源站点的origin（不包括路径）
    • origin-when-cross-origin：对同源请求发送完整的URL，对跨源请求只发送origin
    • same-origin：仅对同源请求发送Referer头部
    • strict-origin：仅在协议安全级别相同的情况下发送origin
    • strict-origin-when-cross-origin：对同源请求发送完整的URL，跨源请求中，只有当协议安全级别相同才发送origin

Man-in-the-middle-attack 中间人攻击​

  • 攻击者插入自己到客户端和服务器之间的通信中。攻击者可以拦截、修改或者伪造数据包。对于HTTP协议的中间人攻击，攻击者可以轻松地查看明文传输的数据，如密码和敏感信息
  • 使用HTTPS协议，它通过SSL/TLS加密数据。即使数据包经过攻击者，由于加密，攻击者也无法轻易查看到内容。此外，还可以通过在服务器端添加CA证书，并确保客户端能够验证这些证书的有效性来增加安全性

需要总结的就是整个项目的前后台一体化的技术栈选择思路，前后台项目中的常见问题以及原因，解决方案，还有大型项目的协同开发，项目分解，项目开发的优化问题；比如复杂数据的处理，大量数据的加载性能；大型项目的优化拆解，例如微前端思想, 深入理解下浏览器的工作原理

浏览器的渲染和资源解析过程：​

  • 浏览器从上到下解析HTML文档，生成DOM节点树。
  • 在遇到样式时，浏览器会异步下载，并构建CSSOM树。
  • 当遇到不带 async 和 defer 的 script 时，浏览器会暂停解析HTML，进行下载和执行。
  • CSS和DOM渲染、JS和DOM解析之间存在阻塞关系。
  • 布局（Layout）和绘制（Paint）分别涉及将DOM节点布局到屏幕的正确位置和绘制所有节点到屏幕上

优化浏览器渲染：​

  • 将JavaScript放在body最后，使用异步加载JS资源。
  • 优化CSS加载，使用内嵌样式或link标签而非 @import
  • 减少回流重绘，避免频繁操作样式和DOM，使用transform代替复杂动画

处理Cookies和Session：​

  • Cookies可以通过HTTP请求头或JavaScript设置
  • Session比Cookie安全，存储在服务器端
  • Cookies主要用于存储用户信息，而Session通常用于存储登录信息等重要信息

Web安全：​

  • 防范跨站脚本攻击（XSS），使用三方库如dompurify来过滤可能的攻击代码
  • 防范跨站请求伪造（CSRF），通过添加referer头字段或使用token验证
  • 防范SQL注入，过滤SQL语句中的关键词
  • 防范分布式拒绝服务攻击（DDoS），使用高防服务器、黑名单策略、DDoS清洗等

浏览器输入URL后的过程：​

  • URL解析、DNS查询、TCP三次握手、发送HTTP请求、响应请求、页面渲染、四次挥手

网络和缓存处理：​

  • 处理重定向、应用缓存（例如通过强缓存和协商缓存）
  • DNS域名解析、TCP/IP协议、HTTP发展与特点
  • HTTPS的7次握手过程

JavaScript的单线程性：​

  • JS是单线程的，主要是为了简化DOM操作和避免复杂的同步问题。HTML5引入了Web Worker标准，允许创建多个线程，但这些线程不能操作DOM并受主线程控制

CSS与DOM的关系：​

  • CSS不会阻塞DOM的解析，但会阻塞DOM的渲染
  • CSS会阻塞后续JS的执行，但不会阻塞JS文件的下载

JavaScript对页面的影响：​

  • JS会阻塞DOM的解析，因此也会影响页面的加载
  • 常见的做法是将JS文件放在页面底部
